# Durable Navivox connection credentials

Navivox will treat Pairing handoff tokens as short-lived bootstrap credentials, not as persisted connection secrets. Durable Android reconnect will use a Gormes-issued, revocable device credential bound to the authenticated Gateway identity and a Navivox-generated random non-secret app install identity. Issuance happens immediately after authenticated connect plus capability/transport proof, before chat, and success is shown as a small non-secret notice/settings status rather than a modal; credential IDs are hidden in normal UI and only redacted/truncated in advanced diagnostics/export; if issuance fails, Pairing handoff still completes because Navivox connected successfully, and Navivox may continue session-only while showing that reconnect was not saved. Navivox may perform one bounded automatic retry for transient issuance failure, then leaves manual retry to gateway/settings UI; manual retry may use existing active session auth when still connected and allowed, otherwise it requires a new Pairing handoff. The gold-standard credential uses one keypair per Gormes gateway and Gormes-issued short-TTL single-use nonce challenges, signing canonical JSON bytes rather than ad-hoc concatenated strings. REST reconnect uses a short-lived, non-persisted access token minted through the challenge flow, and WebSocket reconnect uses a short-lived stream session token minted through the same REST challenge path rather than signing directly in WebSocket subprotocol/header constraints. Durable credential issuance and reconnect may use plain HTTP only for loopback/local bridge; non-loopback exposure requires HTTPS or an authenticated private-network posture such as Tailscale/WireGuard advertised by the gateway. Navivox refuses durable credential issuance/reconnect by default on insecure non-loopback HTTP instead of allowing warning-only persistence. Android/Termux same-device setup uses the authenticated Pairing handoff `base_url`/`websocket_url` as connection authority; Navivox does not assume app loopback reaches Termux loopback or rewrite transport identity without authenticated gateway confirmation. Descriptor URLs are only preflight input for obvious-insecure checks; final durable credential eligibility and transport-security classification come from authenticated status/capabilities. The gateway should expose detailed transport-security fields plus a simple effective enum/gate so Navivox has one clear persistence decision and useful diagnostics. First-slice access/session token invalidation uses short TTL plus underlying credential revocation, not individual token revocation; REST access tokens default to about 5 minutes, and WebSocket stream tokens are single-connection-attempt tokens with no more than about 60 seconds to open the stream; implementation should go straight to ECDSA P-256 with SHA-256 keypair challenge through a small Android-native keystore MethodChannel for non-exportable private-key generation/signing if Android secure key APIs are feasible from Flutter/native bridge, with RSA-PSS fallback only if Android compatibility testing forces it; Navivox sends public keys to Gormes as JWK payloads with algorithm metadata such as `kty`, `crv`, `x`, `y`, and `alg`; any bearer-token device credential is explicitly interim and stored only in platform secure storage under an alias based on authenticated Gateway identity when available, with only a temporary local record ID before proof. Broad secure-storage dependencies are avoided for the first Android keypair slice unless the native bridge proves impractical, and future iOS support should use a parallel Keychain/Secure Enclave adapter. Gateway identity is opaque-public: safe as an identifier but not human-readable and not derived from machine/user/path data. The install identity and other non-secret gateway metadata may be stored separately in shared preferences, and key aliases never include base URLs, tokens, usernames, machine names, display labels, or paths. Durable credential metadata belongs to gateway auth/admin state and local gateway metadata, never Gormes memory, Profile contact state, Transcript surface, sessions, run records, or agent-visible records. Flutter web will not get silent durable reconnect initially, and reconnect must prove the same Gateway identity before updating local metadata.

## Considered Options

- Persist Pairing handoff tokens: rejected because bootstrap tokens are meant for first-run transfer and are too easy to leak through shared text, screenshots, routes, or logs.
- Store bearer secrets in shared preferences or browser storage: rejected because those stores are not an acceptable secret boundary for durable gateway access.
- Require a new Pairing handoff after every Android restart: rejected because it harms the connect-and-talk loop when platform secure storage can protect a device credential.

## Consequences

Gormes needs credential scope storage plus a minimal authenticated device credential issue/list/revoke API; first-slice Navivox client access is capability-gated and scoped to its own install, while Gormes CLI/admin may list/revoke all installs. Credential issuance lives on a separate mutating endpoint rather than `/v1/navivox/status`. The issuance endpoint accepts Pairing handoff token auth for initial issuance/re-pair and existing durable credential auth only for self-rotation of that install. Self-rotation re-evaluates current gateway policy and caps new scopes to the previous credential scopes so rotation cannot expand privileges, while Pairing handoff re-pair can issue current default scopes; revocation must block future auth immediately. Reinstalling Navivox creates a new app install identity and therefore a new credential; old credentials remain server-revocable but are not treated as the same physical device or OS user. A Gormes config reset invalidates durable credentials unless the operator explicitly preserves or restores both Gateway identity and the credential registry during backup or migration. Normal backup/export may preserve opaque-public Gateway identity by default for same-gateway restore, but credential registry export should require an explicit “preserve paired Navivox devices” mode. If a restore preserves Gateway identity but not the credential registry, Navivox keeps non-secret metadata and treats it as a known gateway that requires a new Pairing handoff, not as a different gateway; credential failure does not auto-delete gateway metadata. A new Pairing handoff may replace an existing credential automatically only after authenticated proof of the same Gateway identity; there is one active credential per Gateway identity plus app install identity, so rotation/re-pair replaces the prior active grant. Gormes may keep redacted audit history for revoked credentials, including last-used timestamps, but diagnostic export omits IP addresses, user agents, and device fingerprints by default; active secret/key material and bearer tokens must be removed from the auth registry and never retained as audit payload. Credential labels default to bland copy like “Navivox Android” and should be operator-editable rather than auto-derived from model, username, account, hostname, or other device identifiers. A different authenticated Gateway identity creates or updates a separate gateway record rather than replacing the current one, and switching to it requires confirmation when another gateway is active. Navivox `forget gateway` should attempt remote revoke for only this app install credential by default, but always remove local credential and metadata, reporting remote revocation as unconfirmed if Gormes is unreachable. Normal UI does not expose a separate local-only forget action because it invites leftover server grants; all-device revoke is reserved for Gormes CLI/admin or future management UI. The first durable credential slice should include Gormes CLI emergency commands to revoke this install, revoke by credential ID, and revoke all Navivox credentials for the gateway, with confirmation prompts and `--yes` support for scripted exact/broad revokes.
