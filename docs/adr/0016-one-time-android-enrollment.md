# ADR 0016: Enroll Android with one-time pairing codes

Status: accepted
Date: 2026-07-13

Android enrollment uses short-lived, single-use pairing codes. A QR or `wing://connect` payload identifies the normalized endpoint, pairing code, and optionally a same-host Wing CLI broker. After reviewing the authority's label, expiry, and actual access grant, Android exchanges the code and stores the returned credential in platform secure storage. Hermes Agent can issue a scoped operator token; for existing `API_SERVER_KEY` servers, `wing-cli` can instead expose that configured key once from a temporary Tailscale-hosted broker without changing Hermes Agent.

## Consequences

- Bearer tokens and compatibility superuser keys never appear in URLs, QR payloads, Android shared text, logs, screenshots generated by the app, or clipboard flows.
- The selected enrollment authority owns code creation, expiry, and single-use redemption. Hermes Agent owns scoped-token issuance; the CLI compatibility broker only relays the already-configured superuser key and exits after one exchange or timeout.
- Hermes Wing shows the endpoint and requested scopes before redemption and fails closed on malformed, expired, reused, or origin-mismatched payloads.
- Existing Android connect-intent parsing is ingress only; Dart performs strict payload validation and the authenticated exchange.
- Manual scoped-token entry remains a developer fallback. Manual superuser-key entry requires an explicit warning; the CLI broker is the preferred compatibility path when Hermes Agent has only `API_SERVER_KEY` authentication.
